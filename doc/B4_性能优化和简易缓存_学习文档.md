# B4: 性能优化和简易缓存 — 详细学习文档

## 一、阶段目标

B4 阶段的核心目标是通过指令缓存 (ICache) 和性能计数器 (PFC) 优化处理器性能，建立数据驱动的优化方法论。完成后你将：

1. 理解缓存的基本概念（局部性原理、命中/缺失、AMAT）
2. 实现 W-Way 组相联指令缓存 (ICache)
3. 实现 `fence.i` 指令维护缓存一致性
4. 搭建性能计数器框架，统计 IPC/CPI/Hit Rate/AMAT
5. 用数据指导优化方向（Amdahl 定律的工程实践）
6. 理解 ICache 与 AXI4 总线的交互（burst 填充）
7. 掌握缓存的形式验证方法

## 二、B4 与前置阶段的关系

### 2.1 B3 分析告诉了我们什么？

B3 的时序分析结果揭示了性能瓶颈：
- 每次取指都需要多周期 AXI 事务（3+ cycles）
- CPI >> 1，主要时间花在等待内存
- 需要减少内存访问延迟

### 2.2 为什么需要 ICache？

| 无 ICache | 有 ICache |
|-----------|-----------|
| 每条指令都要通过 AXI 从内存读取 | 命中时 1 cycle 返回指令 |
| CPI ≈ 3-5 (受 AXI 延迟支配) | CPI 接近 1 (高命中率时) |
| 总线带宽成为瓶颈 | 只有 miss 时才访问总线 |

**量化分析**（假设 AXI 读延迟 = 5 cycles）:
- 无 ICache: CPI = 5 (每条指令等 5 cycles)
- 有 ICache (95% hit rate): CPI = 0.95×1 + 0.05×5 = 1.20
- 性能提升: 5/1.20 = **4.2 倍**

### 2.3 B1/B2 为 ICache 准备了什么？

- **AXI4 burst 支持** (B2): ICache miss 时可以一次 burst 读取整个 cache line
- **AXI Demux** (B1/B2): IFU 的请求可以按地址路由到 ICache 或直接内存
- **Valid/Ready 握手**: ICache miss 时 IFU 自然阻塞，不需要额外 stall 逻辑

## 三、缓存基础概念

### 3.1 局部性原理

缓存之所以有效，是因为程序访问具有**局部性**：

| 类型 | 含义 | 指令取指的体现 |
|------|------|----------------|
| **时间局部性** | 最近访问的数据很快还会再访问 | 循环体中的指令反复执行 |
| **空间局部性** | 访问某地址后，附近地址很快被访问 | 顺序执行（PC+4） |

### 3.2 缓存术语

| 术语 | 英文 | 含义 |
|------|------|------|
| **Cache Line** | — | 缓存的最小存储单位 (本项目: 8 字节) |
| **Hit** | 命中 | 请求的数据在缓存中找到 |
| **Miss** | 缺失 | 请求的数据不在缓存中，需要从内存加载 |
| **Hit Rate** | 命中率 | 命中次数 / 总访问次数 |
| **Miss Penalty** | 缺失代价 | 从发现 miss 到数据可用的周期数 |
| **AMAT** | Average Memory Access Time | 平均访存时间 |
| **Tag** | 标签 | 用于识别 cache line 存的是哪个地址的数据 |
| **Index** | 索引 | 确定地址映射到哪个 cache set |
| **Offset** | 偏移 | cache line 内的字节偏移 |

### 3.3 AMAT 公式

```
AMAT = Hit Time + Miss Rate × Miss Penalty

其中:
  Hit Time    = 缓存命中时的访问时间 (本项目: 1-2 cycles)
  Miss Rate   = 1 - Hit Rate
  Miss Penalty = 从内存加载 cache line 的时间 (本项目: 5-10 cycles)
```

### 3.4 地址分解

```
         31        TAG_LSB  INDEX_MSB   INDEX_LSB  OFFSET_MSB  0
         ┌──────────────┬──────────────┬────────────────────────┐
  地址:  │     Tag      │    Index     │        Offset          │
         └──────────────┴──────────────┴────────────────────────┘
         
本项目参数:
  CACHELINE_SIZE = 8 字节       → Offset 位宽 = log2(8) = 3 bits
  CACHELINE_NUM  = 16           → 每路 8 行 (16/2way = 8 sets)
  CACHEWAY_NUM   = 2 (2路组相联) → Index 位宽 = log2(8) = 3 bits
  Tag 位宽 = 32 - 3 - 3 = 26 bits
```

## 四、缓存组织结构

### 4.1 直接映射 vs 组相联 vs 全相联

| 类型 | 映射规则 | 冲突 | 硬件复杂度 |
|------|----------|------|-----------|
| 直接映射 | 每个地址只能放在一个位置 | 高 (同 index 必冲突) | 低 |
| **W-Way 组相联** | 同 index 可放在 W 个位置 | 中 (同 set 的 W 路满了才冲突) | 中 |
| 全相联 | 可放在任何位置 | 低 | 高 (需要 CAM) |

### 4.2 本项目: 2-Way 组相联

```
         Set 0   Set 1   Set 2   Set 3   Set 4   Set 5   Set 6   Set 7
Way 0:  [line0] [line1] [line2] [line3] [line4] [line5] [line6] [line7]
Way 1:  [line0] [line1] [line2] [line3] [line4] [line5] [line6] [line7]
         
每个 line 包含:
  ├── valid (1 bit)     : 有效位
  ├── tag   (26 bits)   : 标签
  └── data  (8 bytes)   : 数据 (= 2 条 32-bit 指令)

总容量: 8 sets × 2 ways × 8 bytes = 128 字节
```

### 4.3 查找过程

```
输入地址: 0xa000_0104

1. 分解地址:
   Tag    = 0xa000_0104[31:6] = 0x2800004
   Index  = 0xa000_0104[5:3]  = 0 (第0个set)
   Offset = 0xa000_0104[2:0]  = 4 (line内偏移4字节)

2. 用 Index 找到 Set 0:
   Way 0: valid=1, tag=0x2800004  → TAG匹配! HIT!
   Way 1: valid=1, tag=0x2800008  → 不匹配

3. 用 Offset 从 Way 0 的 data 中取出第 4~7 字节

4. 返回: 32-bit 指令
```

## 五、ICache 实现详解 (`axi_icache.v`)

### 5.1 参数配置

```verilog
localparam CACHELINE_SIZE = 8;    // 每行 8 字节 (2 条指令)
localparam CACHELINE_NUM  = 16;   // 总行数 16
localparam CACHEWAY_NUM   = 2;    // 2 路组相联

// 自动计算的参数:
localparam CACHELINE_SIZE_WIDTH  = $clog2(CACHELINE_SIZE);      // = 3 (offset 位宽)
localparam CACHELINE_INDEX_WIDTH = $clog2(CACHELINE_NUM / CACHEWAY_NUM); // = 3 (index 位宽)
localparam TAG_LSB   = CACHELINE_SIZE_WIDTH + CACHELINE_INDEX_WIDTH;     // = 6
localparam INDEX_MSB = TAG_LSB - 1;                                       // = 5
localparam INDEX_LSB = CACHELINE_SIZE_WIDTH;                              // = 3
localparam OFFSET_MSB = INDEX_LSB - 1;                                    // = 2
```

### 5.2 存储结构

```verilog
// 数据存储: [sets数][ways数] 的二维数组, 每项存一个 cache line
reg [CACHELINE_SIZE*8-1:0] data_array [CACHELINE_NUM/CACHEWAY_NUM-1:0][CACHEWAY_NUM-1:0];
// 即: reg [63:0] data_array [7:0][1:0];  // 8 sets × 2 ways × 64 bits

// 有效位: 每个 set 有 W 个 valid bit (每个 way 一个)
reg [CACHEWAY_NUM-1:0] data_valid [CACHELINE_NUM/CACHEWAY_NUM-1:0];
// 即: reg [1:0] data_valid [7:0];  // 8 sets, 每 set 2 个 valid bit

// 标签存储: 与 data_array 同构
reg [`CPU_WIDTH-1:TAG_LSB] tag_array [CACHELINE_NUM/CACHEWAY_NUM-1:0][CACHEWAY_NUM-1:0];
// 即: reg [31:6] tag_array [7:0][1:0];  // 8 sets × 2 ways × 26 bits tag
```

### 5.3 状态机

```
┌──────┐      ┌──────────────┐      ┌──────────────┐
│ idle │─────▶│ check_cache  │─────▶│ instr_fetch  │──▶ (返回idle)
└──────┘      └──────────────┘      └──────────────┘
   ↑  接收       │  HIT↗              ↑ 数据就绪
   │  AR请求     │                     │
   │             │ MISS↘               │
   │             ▼                     │
   │       ┌──────────────┐      ┌──────────────┐      ┌──────────────┐
   │       │ mem_araddr   │─────▶│  mem_rdata   │─────▶│ update_array │
   │       └──────────────┘      └──────────────┘      └──────────────┘
   │        发出burst请求          接收burst数据         更新完毕
   └─────────────────────────────────────────────────────────────────────
```

各状态的作用:

| 状态 | 功能 | 持续周期 |
|------|------|----------|
| `idle` | 等待 IFU 的 AR 请求 | 1 cycle |
| `check_cache` | 读取 tag/valid，判断 hit/miss | 1 cycle |
| `instr_fetch` | 命中时从 data_array 读取数据返回 | 1 cycle |
| `mem_araddr` | miss 时发出 AXI burst 读请求 | 1-2 cycles |
| `mem_rdata` | 接收 burst 数据，逐拍写入 cache | N cycles (burst长度) |
| `update_array` | 过渡状态 | 1 cycle |

### 5.4 命中检查函数

```verilog
function [CACHEWAY_WIDTH:0] check_cache(input [`CPU_WIDTH-1:TAG_LSB] cache_tag);
  integer i;
  for (i = 0; i < CACHEWAY_NUM; i = i + 1) begin
    if (icache_valid[i] && cache_tag == icache_tag[i]) begin
      return {i, 1'b1};    // 返回 {way编号, 命中标志}
    end
  end
  return 0;                  // 未命中
endfunction
```

**返回值**: `{way_sel, hit}` — 低 1 位是命中标志，高位是命中的 way 编号。

### 5.5 Miss 处理 — AXI4 Burst 填充

```verilog
// mem_araddr 状态: 发起 burst 读请求
icache_araddr_next = ifu_read_addr_reg & ~{(OFFSET_MSB+1){1'b1}}; // 对齐到 line 起始
icache_arlen_next  = CACHELINE_SIZE / (1 << ifu_read_size_reg) - 1; // burst 长度
// CACHELINE_SIZE=8, size=2(4字节) → arlen = 8/4 - 1 = 1 (2拍 burst)

// mem_rdata 状态: 逐拍接收数据写入 cache
if (icache_rvalid && icache_rready_reg) begin
  data_wr_en = 1'b1;
  icache_wr_offset_next = icache_wr_offset_reg + (1 << icache_arsize_reg);
  if (icache_rlast) begin  // 最后一拍
    state_next = update_array_t;
  end
end
```

**地址对齐**: `addr & ~{3{1'b1}}` 将低 3 位清零，对齐到 8 字节 cache line 边界。

### 5.6 替换策略

```verilog
// miss 时选择替换哪一路:
icache_way_sel_next = icache_way_sel_reg + 1'b1;  // 简单轮转
```

当前实现使用**简单轮转 (Round-Robin)** 替换策略。每次 miss 时选择下一个 way。

更复杂的替换策略（LRU/Random）可以进一步减少冲突 miss，但硬件开销更大。

## 六、fence.i — 缓存一致性

### 6.1 为什么需要 fence.i？

ICache 只缓存指令。如果程序通过 Store 指令修改了内存中的代码（自修改代码），ICache 中的旧指令就过期了。`fence.i` 指令确保后续取指能看到最新的内存内容。

**典型场景**:
- Bootloader 将代码从 Flash 复制到 SDRAM → 需要 fence.i
- 动态代码生成（JIT 编译）→ 需要 fence.i
- OS 加载新程序 → 需要 fence.i

### 6.2 fence.i 的硬件实现

```verilog
// 收到 fence.i 信号时，无效化所有 cache line:
always @(posedge i_clk) begin
  if (fence_i) begin
    for (i = 0; i < CACHELINE_NUM / CACHEWAY_NUM; i = i + 1) begin
      data_valid[i] <= {CACHEWAY_NUM{1'b0}};  // 所有 valid 位清零
    end
  end
end
```

**实现策略**: 全部无效化 (Invalidate All)。这是最简单的实现：
- 优点: 硬件极简（只需清零 valid 位）
- 缺点: fence.i 后所有行都需要重新从内存加载（cold miss 增加）
- 适用: 对于 ICache (只读)，这是 RISC-V 规范推荐的方式

### 6.3 fence.i 在流水线中的传递

```
IDU 译码出 fence.i 指令
    │
    ▼
top.v: fence_i 信号连接到 axi_icache 的 fence_i 输入
    │
    ▼
ICache: 清零所有 valid 位
    │
    ▼
后续取指: 全部 miss → 从内存重新加载 (保证一致性)
```

## 七、性能计数器 (Performance Counters)

### 7.1 为什么需要性能计数器？

**"You can't optimize what you can't measure."** — 没有数据就不知道瓶颈在哪。

性能计数器通过 DPI-C 在仿真时统计各种指标，指导优化方向。

### 7.2 统计的指标

| 指标 | 含义 | 计算方式 |
|------|------|----------|
| **Total cycles** | 总时钟周期数 | 仿真时间 / 2 |
| **Total insts** | 总指令数 | 指令提交计数 |
| **IPC** | 每周期指令数 | insts / cycles |
| **CPI** | 每指令周期数 | cycles / insts |
| **IFU Read PFC** | IFU 读事务次数 | AXI AR/R 握手计数 |
| **LSU Read/Write PFC** | LSU 读写事务次数 | AXI 握手计数 |
| **IDU Cal/Mem/Jump/CSR** | 各类指令占比 | 按类型分类计数 |
| **Per-type CPI** | 各类指令的平均 CPI | 该类总周期 / 该类指令数 |
| **ICache Hit Rate** | 缓存命中率 | hits / total_accesses |
| **ICache AMAT** | 平均访存时间 | hit_time + miss_rate × miss_penalty |
| **ICache TMT** | 真实缺失时间 | miss_rate × miss_penalty |
| **LSU Read/Write Delay** | 平均读写延迟 | 总延迟 / 事务数 |

### 7.3 RTL 端实现 (ICache 内)

```verilog
`ifndef SYNTHESIS
  // 仿真时统计缓存性能 (不参与综合)
  reg [31:0] cache_hit_rate;   // 命中次数
  reg [31:0] cache_acc_tot;    // 总访问次数
  reg [31:0] access_time;      // 总访问周期数
  reg [31:0] miss_penalty;     // 总缺失周期数

  // 统计命中次数
  always @(posedge i_clk) begin
    if (!i_rst_n)
      cache_hit_rate <= 0;
    else if (state_reg == check_cache_t && check_cache(ifu_cache_tag))
      cache_hit_rate <= cache_hit_rate + 1;
  end

  // 统计总访问次数
  always @(posedge i_clk) begin
    if (!i_rst_n)
      cache_acc_tot <= 0;
    else if (ifu_arvalid && ifu_arready_reg)
      cache_acc_tot <= cache_acc_tot + 1;
  end

  // 通过 DPI-C 导出给 C++ 分析
  import "DPI-C" function void cache_AMAT(
    input int hit_rate, input int acc_tot,
    input int access_time, input int miss_penalty
  );
  always @(*) cache_AMAT(cache_hit_rate, cache_acc_tot, access_time, miss_penalty);
`endif
```

### 7.4 C++ 端实现 (`main.cpp`)

```cpp
// 仿真结束时输出性能报告:

printf("Total insts: %ld\n", total_inst);

// 按指令类型分析:
printf("IDU Cal: %ld, %.2f%%, CPI: %f\n", idu_cal_type, ...);

```

### 7.5 AXI 握手计数器 (`dpic.cpp`)

```cpp
extern "C" void axi4_handshake(svBit valid, svBit ready, svBit last, int pfc_type) {
  if (valid && ready && last) {
    switch (pfc_type) {
      case 1: ifu_pfc_r++; break;                    // IFU 读完成
      case 2: case 5: lsu_begin_time = get_sim_time(); break; // LSU 开始
      case 4: lsu_pfc_w++; lsu_write_delay += ...; break;     // LSU 写完成
      case 6: lsu_pfc_r++; lsu_read_delay += ...; break;      // LSU 读完成
    }
  }
}
```

## 八、ICache 与系统集成

### 8.1 在 top.v 中的位置

```
IFU (AXI4 Master)
  │
  ▼
AXI Demux (地址分路)
  │              │
  │ port[0]      │ port[1]
  ▼              ▼
ICache       直接内存路径
  │
  ▼
AXI4 Interconnect → 主存储器
```

**AXI Demux 的地址规则**:
- SDRAM 地址范围 (0xa0000000~0xbfffffff) → port[0] → **经过 ICache**
- Flash/SRAM 地址 → port[1] → **绕过 ICache** (启动阶段不缓存)

### 8.2 接口设计

ICache 模块有两组 AXI4 接口：
- **上游 (IFU 侧)**: 接收 IFU 的取指请求，单拍返回或阻塞
- **下游 (Memory 侧)**: miss 时发出 burst 请求加载 cache line

```verilog
axi_icache u_axi_icache (
  .i_clk(clock), .i_rst_n(rst_n_sync),
  .fence_i(fence_i),           // 来自 IDU 的一致性信号
  
  // IFU 侧 (上游):
  .ifu_arvalid(ifu2icache_axi_arvalid),
  .ifu_araddr (ifu2icache_axi_araddr),
  .ifu_rdata  (ifu2icache_axi_rdata),
  .ifu_rvalid (ifu2icache_axi_rvalid),
  // ...
  
  // Memory 侧 (下游):
  .icache_arvalid(icache_axi_arvalid),
  .icache_araddr (icache_axi_araddr),
  .icache_arlen  (icache_axi_arlen),    // burst 长度
  .icache_rdata  (icache_axi_rdata),
  .icache_rlast  (icache_axi_rlast),    // burst 最后一拍
  // ...
);
```

### 8.3 ICache 的写通道处理

ICache 是**只读缓存**（只缓存指令），所以写通道全部禁用：

```verilog
// IFU 侧写通道: IFU 不写指令
assign ifu_awready = 1'b0;
assign ifu_wready  = 1'b0;
assign ifu_bvalid  = 1'b0;

// Memory 侧写通道: ICache 不写内存
assign icache_awvalid = 1'b0;
assign icache_wvalid  = 1'b0;
assign icache_bready  = 1'b0;
```

## 十、数据驱动的优化方法论

### 10.1 Amdahl 定律

```
加速比 = 1 / ((1 - P) + P/S)

其中:
  P = 可优化部分占总时间的比例
  S = 该部分的加速倍数

例: 取指占 CPI 的 80%, ICache 使取指加速 5 倍:
  加速比 = 1 / ((1 - 0.8) + 0.8/5) = 1 / (0.2 + 0.16) = 2.78 倍
```

### 10.2 PFC 指导优化的流程

```
1. 运行基准测试，收集 PFC 数据
   $ make run IMAGE=coremark.bin
   
2. 分析报告:
   CPU CPI: 4.2
   IDU Cal: 45%, CPI: 1.5
   IDU Mem: 30%, CPI: 8.0    ← 访存指令 CPI 最高!
   IDU Jump: 25%, CPI: 5.0   ← 跳转指令也较高
   ICACHE Hit Rate: 0.92
   ICACHE AMAT: 1.8 cycles

3. 识别瓶颈:
   - 总 CPI = 0.45×1.5 + 0.30×8.0 + 0.25×5.0 = 0.675 + 2.4 + 1.25 = 4.3
   - 访存指令贡献了 2.4/4.3 = 56% 的 CPI
   - ICache 92% hit rate → miss 占 8%, miss penalty 很高
   
4. 决策:
   - 增大 ICache → 减少 miss rate → 降低取指 CPI
   - 或: 添加 DCache → 降低 LSU 读写延迟
   - 或: 分支预测 → 减少 Jump 类指令的 penalty
```

### 10.3 增大 ICache 的效果预测

```
当前: 128B, 2-way, hit rate = 92%, AMAT = 1.8
如果扩大到 256B: hit rate 可能提升到 ~96%
  新 AMAT ≈ 1.0 + 0.04×10 = 1.4  (减少 0.4 cycles per instruction)
  
总 CPI 改善: 约 0.4 × 取指占比(~1.0) = 减少 0.4
  新 CPI ≈ 4.2 - 0.4 = 3.8  (性能提升 ~10%)
```

## 十一、形式验证

### 11.1 ICache 形式验证 (`CacheTest.sv`)

项目提供了 ICache 的形式验证环境：

```
npc/formal/
├── CacheTest.sv    # ICache 测试环境
├── axi_ram.v       # AXI RAM 参考模型
└── Sub.sv          # 子模块验证
```

**验证目标**: 证明 ICache 的行为等价于直接访问内存（透明性）。

```
                  ┌─────────────┐
  IFU 请求 ──────▶│   ICache    │──────▶ IFU 响应 (待验证)
                  └──────┬──────┘
                         │ (miss时)
                         ▼
                  ┌─────────────┐
                  │  AXI RAM    │         参考模型
                  └─────────────┘
                  
验证属性: 对于相同的地址请求, ICache 返回的数据
          必须等于 AXI RAM 中该地址的数据
```

### 11.2 验证的关键属性

- **数据正确性**: hit 时返回的数据 = 内存中该地址的实际内容
- **一致性**: fence.i 后，后续读取反映最新内存状态
- **活性**: 任何请求最终都会得到响应（不会死锁）
- **AXI 协议合规**: valid 不被撤回、burst 序列正确

## 十二、常见问题与解决

### Q1: ICache hit rate 很低 (< 80%)

**可能原因**:
- Cache 太小（总容量 128B 对某些程序不够）
- 程序工作集大于 cache 容量（如大循环体）
- 频繁跳转导致空间局部性差

**解决方向**:
- 增大 `CACHELINE_NUM` 或 `CACHELINE_SIZE`
- 升级为更高关联度（4-way）
- 使用更智能的替换策略（LRU）

### Q2: fence.i 后性能骤降

**原因**: 所有 cache line 无效，后续取指全部 miss（cold start）。
**这是正常行为** — fence.i 的正确性比性能重要。

**优化方向**: 减少 fence.i 的使用频率（只在真正需要时调用）。

### Q3: Cache line 大小如何选择？

| Line Size | 优势 | 劣势 |
|-----------|------|------|
| 4B (1字) | miss penalty 小 | 空间局部性差 |
| **8B (2字)** | 平衡 | 本项目选择 |
| 16B (4字) | 空间局部性好 | miss penalty 大 |
| 32B (8字) | 极好的局部性 | miss 时需要长 burst |

项目选择 8B 是因为：单发射处理器每次只取 1 条指令 (4B)，8B 可以预取下一条。

### Q4: DiffTest 与 ICache 的交互

ICache 是**透明**的——对于 DiffTest 来说，有无 ICache 不影响指令行为，只影响性能。DiffTest 只在指令 commit 时对比寄存器状态，不关心取指用了几个周期。

## 十三、与后续阶段的关系

### 13.1 ICache 为 B5 (流水线) 提供的基础

- ICache hit 时 IFU 可以每周期输出一条指令 → 支持 IPC≈1 的流水线
- ICache miss 时 IFU 自然阻塞 → Valid/Ready 握手处理背压
- ICache 的 burst 填充与流水线解耦 → 不影响其他级的执行

### 13.2 未来可能的优化方向

| 方向 | 说明 | 复杂度 |
|------|------|--------|
| 增大 ICache | 更多行/更大 line | 低 (改参数) |
| 添加 DCache | 数据缓存 (读写) | 高 (写回/写穿策略) |
| 分支预测 | 减少跳转 penalty | 中 |
| 指令预取 | 提前加载下一行 | 中 |
| 非阻塞 Cache | miss 时不完全阻塞 | 高 |

## 十四、PA 讲义相关思考题

### Q: 为什么 ICache 不需要写回 (Write-Back) 策略？

**回答**:
ICache 只缓存指令，CPU 不会对指令地址做 Store 操作（正常执行时）。因此 ICache 的内容永远是内存的副本，不需要写回。当需要一致性时（fence.i），直接无效化即可。

这也是为什么 ICache 比 DCache 简单得多——DCache 需要处理写命中（Write-Through/Write-Back）、脏位、写合并等复杂问题。

---

### Q: 增大 Cache 是否总能提升性能？

**回答**:
不一定。增大 Cache 有递减效应（Diminishing Returns）：
- 128B → 256B: miss rate 可能从 8% 降到 4%（提升明显）
- 256B → 512B: miss rate 可能从 4% 降到 3%（提升减缓）
- 512B → 1KB: miss rate 可能从 3% 降到 2.5%（提升微小）

当 miss rate 已经很低时，再增大 Cache 带来的收益有限（hit time 反而可能增加），而面积成本是线性增长的。

---

### Q: 2-Way 组相联比直接映射好多少？

**回答**:
经验法则：**2:1 缓存规则** — N-way 组相联的效果约等于 2N 容量的直接映射。

即: 2-way 128B ≈ 1-way 256B（在 miss rate 方面）

但 2-way 的额外硬件成本是：
- Tag 比较器 ×2
- 数据 MUX ×2
- 替换逻辑

在面积受限的嵌入式处理器中，2-way 通常是最佳性价比选择。
