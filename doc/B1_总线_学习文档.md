# B1: 总线 — 详细学习文档

## 一、阶段目标

B1 阶段的核心目标是实现标准的 AXI4 片上总线系统，将处理器核与多个外设通过统一的总线互联连接。完成后你将：

1. 理解 AXI4 协议的五通道握手机制
2. 实现 AXI Demux（地址解码分发器）
3. 实现 AXI Arbiter（多 Master 仲裁器）
4. 实现 AXI → AXI-Lite 协议桥
5. 理解地址映射与解码逻辑
6. 将 IFU 和 LSU 从 DPI-C 内存访问迁移到 AXI 总线

---

## 二、为什么需要总线？

### 2.1 问题：单周期处理器中 IFU 和 LSU 如何访问内存/外设？

**D4/C2 阶段的做法**：通过 DPI-C 直接调用 C++ 函数访问主机内存模型——简单但不真实。

**B1 阶段的做法**：通过 AXI4 总线——标准化、可综合、真实硬件行为。

```
之前 (DPI-C):                        现在 (AXI 总线):
  IFU ──DPI-C──▶ C++ 内存              IFU ──AXI──┐
  LSU ──DPI-C──▶ C++ 内存              LSU ──AXI──┤
                                                    ▼
                                            AXI Interconnect
                                                    │
                                         ┌──────────┼──────────┐
                                         ▼          ▼          ▼
                                       SRAM      CLINT       UART
```

### 2.2 总线解决的核心问题

| 问题 | 解决方案 |
|------|----------|
| 多个 Master 竞争同一个 Slave | 仲裁器 (Arbiter) |
| 一个 Master 需要访问不同 Slave | 地址解码 (Demux) |
| 不同设备的接口复杂度不同 | 协议桥 (AXI→AXI-Lite) |
| 需要标准化接口 | AXI4 协议规范 |

## 三、AXI4 协议基础

### 3.1 五个独立通道

AXI4 将读写操作分为 5 个独立的通道，**每个通道都有自己的 valid/ready 握手**：

```
Master                                    Slave
  │                                         │
  │ ──── AW Channel (Write Address) ──────▶ │  写地址通道
  │ ──── W  Channel (Write Data) ─────────▶ │  写数据通道
  │ ◀──── B  Channel (Write Response) ───── │  写响应通道
  │                                         │
  │ ──── AR Channel (Read Address) ───────▶ │  读地址通道
  │ ◀──── R  Channel (Read Data) ──────── │  读数据通道
  │                                         │
```

### 3.2 每个通道的信号

**AW 通道 (Write Address)**：
| 信号 | 方向 | 宽度 | 说明 |
|------|------|------|------|
| `awvalid` | M→S | 1 | 地址有效 |
| `awready` | S→M | 1 | Slave 可接收 |
| `awaddr` | M→S | 32 | 写地址 |
| `awid` | M→S | 4 | 事务 ID |
| `awlen` | M→S | 8 | 突发长度 - 1 |
| `awsize` | M→S | 3 | 每拍字节数 (log2) |
| `awburst` | M→S | 2 | 突发类型 (FIXED/INCR/WRAP) |

**W 通道 (Write Data)**：
| 信号 | 方向 | 宽度 | 说明 |
|------|------|------|------|
| `wvalid` | M→S | 1 | 数据有效 |
| `wready` | S→M | 1 | Slave 可接收 |
| `wdata` | M→S | 32 | 写数据 |
| `wstrb` | M→S | 4 | 字节使能 (每位对应一个字节) |
| `wlast` | M→S | 1 | 突发最后一拍 |

**B 通道 (Write Response)**：
| 信号 | 方向 | 宽度 | 说明 |
|------|------|------|------|
| `bvalid` | S→M | 1 | 响应有效 |
| `bready` | M→S | 1 | Master 可接收 |
| `bresp` | S→M | 2 | 响应码 (OKAY/SLVERR/DECERR) |
| `bid` | S→M | 4 | 事务 ID |

**AR 通道 (Read Address)** — 与 AW 通道类似

**R 通道 (Read Data)**：
| 信号 | 方向 | 宽度 | 说明 |
|------|------|------|------|
| `rvalid` | S→M | 1 | 数据有效 |
| `rready` | M→S | 1 | Master 可接收 |
| `rdata` | S→M | 32 | 读数据 |
| `rresp` | S→M | 2 | 响应码 |
| `rlast` | S→M | 1 | 突发最后一拍 |
| `rid` | S→M | 4 | 事务 ID |

### 3.3 握手规则

```
         ┌───────┐           ┌───────┐
  valid ─┤       ├─── → ────┤       ├─ valid
         │ Master│           │ Slave │
  ready ─┤       ├─── ← ────┤       ├─ ready
         └───────┘           └───────┘

规则:
1. 数据在 valid=1 && ready=1 的时钟上升沿传输
2. valid 一旦拉高，在 ready 拉高之前不能撤回
3. ready 可以先于 valid 拉高 (表示 Slave 随时可接收)
4. valid 和 ready 之间不能有组合逻辑依赖 (避免死锁)
```

**时序图示例** (单次传输)：
```
clock:  ─┐  ┌──┐  ┌──┐  ┌──┐  ┌──┐  ┌──
valid:  ──────────┐     │     │     │
                  └─────────────────────
ready:  ──────────────────┐     │
                          └─────────────
传输:                      ↑ (此刻握手成功)
```

### 3.4 突发传输 (Burst)

单次传输效率低（每个地址要单独握手）。AXI4 支持突发传输：

```
awlen = 3 (4 拍突发)
awsize = 2 (每拍 4 字节)
awburst = INCR (递增地址)

时序:
  AW:  addr=0x100 ──────────────────────────
  W:   data[0] ─── data[1] ─── data[2] ─── data[3](last)
  B:                                         ── OKAY ──

效果: 一次性写入 0x100, 0x104, 0x108, 0x10C 共 16 字节
```

## 四、项目中的 AXI 互联架构

### 4.1 整体拓扑

```
  ┌─────┐         ┌─────┐
  │ IFU │(Master0)│ LSU │(Master1)
  └──┬──┘         └──┬──┘
     │ AXI            │ AXI
     ▼                ▼
  ┌──────────────────────────┐
  │     AXI Interconnect      │
  │  (Demux + Arbiter)        │
  └──┬────────┬────────┬─────┘
     │        │        │
     ▼        ▼        ▼
  ┌──────┐ ┌──────┐ ┌──────┐
  │ SRAM │ │CLINT │ │ UART │  (Slaves)
  └──────┘ └──────┘ └──────┘
```

### 4.2 IFU 的 AXI Demux

IFU 的读请求可能去向不同目标（ICache 或主存）：

```verilog
// npc/vsrc/top.v
axi_demux_intf #(
  .NO_MST_PORTS(2),       // 2 个输出端口
  .MAX_TRANS(8)            // 最多 8 个未完成事务
) u_axi_demux (
  .slv_aw_select_i(ifu_slv_select),  // 地址解码结果选择端口
  .slv_ar_select_i(ifu_slv_select),
  .slv(ifu_slv),                      // 输入 (来自 IFU)
  .mst(ifu_mst)                       // 输出 (去向 ICache 或 Memory)
);
```

### 4.3 地址解码 (`addr_decode.sv`)

```systemverilog
// npc/vsrc/libs/addr_decode.sv
localparam rule_t [NoRules-1:0] addr_map = '{
  '{idx: 1, start_addr: 32'h30000000, end_addr: 32'h40000000},  // Flash → 端口1
  '{idx: 0, start_addr: 32'ha0000000, end_addr: 32'hc0000000},  // SDRAM → 端口0
  '{idx: 1, start_addr: 32'h0f000000, end_addr: 32'h10000000}   // SRAM  → 端口1
};
```

**工作原理**：根据请求地址，查找 `addr_map` 确定应该路由到哪个输出端口。

## 五、AXI-Lite 简化协议

### 5.1 AXI-Lite vs AXI4

| 特性 | AXI4 (Full) | AXI-Lite |
|------|-------------|----------|
| 突发传输 | 支持 (awlen/arlen) | 不支持 (每次只传 1 拍) |
| 数据宽度 | 可配置 | 32-bit 或 64-bit |
| ID 字段 | 有 (支持乱序) | 无 |
| 适用场景 | 高带宽 (内存、DMA) | 低速外设 (UART、GPIO) |

### 5.2 为什么需要 AXI → AXI-Lite 桥？

CLINT 和 UART 等简单外设不需要突发传输和乱序完成。用 AXI-Lite 接口可以大幅简化外设控制器的设计。

```verilog
// npc/vsrc/amba/axi_axil_adapter.v
// 将 AXI4 Full 请求转换为 AXI-Lite 请求
// 处理: 突发展开 (burst → 多次单拍), ID 管理
```

## 六、仲裁器 (Arbiter)

### 6.1 问题：IFU 和 LSU 同时请求访问同一个 SRAM 怎么办？

**解决方案**：Round-Robin 仲裁器 — 公平轮流授予访问权。

```verilog
// npc/vsrc/libs/rr_arb_tree.sv
// Round-Robin 仲裁树
// 输入: 多个请求 (req[N-1:0])
// 输出: 授权 (grant[N-1:0]), 同一时刻最多一个 grant=1
```

### 6.2 仲裁策略

| 策略 | 行为 | 特点 |
|------|------|------|
| 固定优先级 | 低编号 Master 总是优先 | 简单但不公平 |
| Round-Robin | 上次授权的下一个优先 | 公平，本项目使用 |
| 加权 RR | 不同 Master 有不同权重 | 复杂但灵活 |

## 七、IFU 和 LSU 的 AXI Master 接口

### 7.1 IFU 的读操作

IFU 只需要读取指令，不需要写入：

```
1. IFU 在 AR 通道发出读请求: araddr=PC, arlen=0 (单拍), arsize=2 (4字节)
2. 等待 arready=1 表示地址被接受
3. 等待 R 通道返回数据: rvalid=1, rdata=指令
4. IFU 拉高 rready=1 接收数据
5. 一次取指完成
```

### 7.2 LSU 的读写操作

LSU 需要读 (Load) 和写 (Store)：

**Load (lw)：**
```
1. AR 通道: araddr=exu_res, arsize=2 (4字节)
2. 等待 R 通道返回: rdata=内存数据
```

**Store (sw)：**
```
1. AW 通道: awaddr=exu_res, awsize=2
2. W 通道: wdata=rs2, wstrb=4'b1111 (全字节使能), wlast=1
3. 等待 B 通道: bresp=OKAY
```

### 7.3 wstrb (字节使能) 的作用

| 指令 | wstrb | 含义 |
|------|-------|------|
| `sw` | `4'b1111` | 写全部 4 字节 |
| `sh` (对齐到低半字) | `4'b0011` | 只写低 2 字节 |
| `sh` (对齐到高半字) | `4'b1100` | 只写高 2 字节 |
| `sb` (偏移 0) | `4'b0001` | 只写最低字节 |
| `sb` (偏移 1) | `4'b0010` | 只写第 2 字节 |
| `sb` (偏移 2) | `4'b0100` | 只写第 3 字节 |
| `sb` (偏移 3) | `4'b1000` | 只写最高字节 |

## 八、从 DPI-C 迁移到 AXI

### 8.1 迁移前 (单周期 DPI-C)

```verilog
// IFU 取指: 组合逻辑瞬时完成
import "DPI-C" function void pmem_read(input int addr, output int data, input bit ren);
```

### 8.2 迁移后 (AXI 总线)

```verilog
// IFU 取指: 需要多个时钟周期
// 周期1: 发出 AR 请求 (arvalid=1)
// 周期2: 等待 Slave 接受地址 (arready=1)
// 周期3~N: 等待数据返回 (rvalid=1)
// 最后: 接收数据 (rready=1, 取指完成)
```

**关键变化**：取指不再是瞬时的！IFU 需要一个**状态机**来管理 AXI 握手：

```verilog
// IFU 状态机 (简化)
localparam IDLE = 0, WAIT_ARREADY = 1, WAIT_RVALID = 2;
reg [1:0] state;

always @(posedge clk) begin
  case (state)
    IDLE: begin
      arvalid <= 1;
      state <= WAIT_ARREADY;
    end
    WAIT_ARREADY: begin
      if (arready) begin
        arvalid <= 0;
        state <= WAIT_RVALID;
      end
    end
    WAIT_RVALID: begin
      if (rvalid) begin
        instr <= rdata;    // 保存指令
        rready <= 1;
        state <= IDLE;     // 完成
      end
    end
  endcase
end
```

### 8.3 Valid/Ready 流水线握手

由于取指/访存现在需要多周期，流水线各级之间需要用 valid/ready 信号做**背压**：

```
  IFU 完成取指后拉高 ifu_valid
  IDU 准备好接收后拉高 idu_ready
  只有 ifu_valid && idu_ready 时数据才传递给 IDU
```

这就是 C 阶段单周期 → B 阶段多周期的核心变化。

```verilog
// npc/vsrc/amba/axi_access_fault.v
// 监控 AXI 响应，如果 bresp/rresp != OKAY 则报告访问异常
```

## 十、常见问题与解决

### Q1: AXI 握手死锁

**症状**：仿真卡死不前进

**原因**：Master 等 ready，Slave 等 valid，形成循环等待

**规则**：`valid` 不能依赖 `ready` 产生（可以先于 ready 拉高）

### Q2: 突发传输时 wlast 没拉高

**症状**：Slave 一直等待更多数据

**解决**：对于单拍传输 (len=0)，`wlast` 必须在唯一的数据拍拉高

### Q3: 读响应的 rlast 处理

对于单拍读取 (arlen=0)，Slave 返回的唯一数据拍必须 `rlast=1`

### Q4: AXI ID 的用途

ID 允许 Slave **乱序**返回响应。如果不需要乱序，所有事务使用相同 ID 即可。本项目中 IFU 和 LSU 使用不同的 ID，方便 Interconnect 路由响应。

## 十一、PA 讲义相关思考题

#### Q: 为什么 AXI 将读和写分为独立通道？

**回答**：
1. **并行性**：读和写可以同时进行（全双工）
2. **解耦**：写地址和写数据可以不同步到达（AW 可以先于 W）
3. **灵活性**：不同通道可以有不同的缓冲深度
4. **乱序**：读响应可以乱序返回（通过 RID 匹配）

---

#### Q: 为什么简单外设用 AXI-Lite 而不直接用 AXI4？

**回答**：
- UART 一次只读写 1 个字节，不需要突发
- 外设控制器面积敏感，AXI-Lite 省掉了 ID、len、burst 等信号
- AXI-Lite 的状态机极简（没有突发计数、没有 ID 管理）
- 通过桥接器可以无缝对接 AXI4 总线

---

#### Q: 引入 AXI 总线后 CPI 会变成多少？

**回答**：CPI > 1。因为：
- 取指至少需要 2-3 个周期（AR 握手 + R 返回）
- Load 指令需要额外 2-3 个周期（AR/AW 握手 + R/B 返回）
- 如果 IFU 和 LSU 同时请求，仲裁会增加等待

典型单发射多周期处理器的 CPI 可能在 3-5 之间。这就是后续 B4 (Cache) 和 B5 (流水线) 要优化的目标。

## 十二、学习建议

1. **先理解 AXI 握手** — 画时序图，确保理解 valid/ready 的含义
2. **从 IFU 开始** — IFU 只有读操作，比 LSU 简单
3. **用简单的 AXI SRAM Slave 测试** — 确认 Master 接口正确
4. **再接入 Arbiter 和 Demux** — 逐步增加复杂度
5. **保留 DPI-C 作为后备** — 开发期间可切换回 DPI-C 快速验证功能正确性

### 13.2 v1.0 (`32fd6d1`) — AXI-Lite 内存接口

**核心变化**：IFU 和 LSU 从 DPI-C 调用改为 AXI-Lite 握手协议。

**IFU 实现** (v1.0 的 `ifu.v`)：
```verilog
module ifu (
  input  i_clk, i_rst_n,
  input  [`CPU_WIDTH-1:0] i_pc,
  output [`INS_WIDTH-1:0] o_instr,
  output o_post_valid,     // 取指完成信号
  input  i_post_ready      // IDU 准备好接收
);
  // AXI-Lite AR 通道状态机:
  // 1. i_post_ready=1 时发起读请求 (arvalid=1)
  // 2. arready=1 时地址被接受
  // 3. rvalid=1 时数据返回，取指完成
  
  always @(*) begin
    arvalid_next = arvalid_reg && !arready;
    if (r_done_reg && ren)
      arvalid_next = 1'b1;  // 发起新的读请求
  end

  // 内置一个 AXI-Lite SRAM Slave
  axi_lite_sram u_axi_lite_sram (...);

  assign o_instr      = rdata;
  assign o_post_valid = rvalid && rready;  // 读数据握手成功 = 取指完成
endmodule
```

**AXI-Lite SRAM Slave** (v1.0 的 `axi_lite_sram.v`)：
```verilog
module axi_lite_sram (...);
  // 读操作状态机:
  // arvalid && !arready && (!rvalid || rready) 时:
  //   1. 调用 DPI-C 读内存: rtl_pmem_read(araddr, rdata_next, 1)
  //   2. 下一拍: arready=1 + rvalid=1 + rdata=读取结果
  //
  // 写操作状态机:
  // awvalid && wvalid && (!awready && !wready) && (!bvalid || bready) 时:
  //   1. 调用 DPI-C 写内存: rtl_pmem_write(awaddr, wdata, wstrb, 1)
  //   2. 下一拍: awready=1 + wready=1 + bvalid=1
endmodule
```

**关键设计决策**：
- SRAM Slave 内部仍用 DPI-C 访问主机内存（仿真阶段必要）
- 但 CPU 侧已通过标准 AXI-Lite 协议通信
- 取指从"瞬时"变为"需要等待一拍" → CPI 从 1 变为 ≥2

## 十四、详解：如何让 NPC 支持 AXI4-Lite（逐步实现）

> 本节基于项目实际代码（`076d790` → `32fd6d1`），详细展示从 DPI-C 到 AXI-Lite 的完整改造过程。

---

### 14.1 改造前的 IFU（DPI-C 版本，commit `076d790`）

```verilog
module ifu (
  input  [`CPU_WIDTH-1:0] i_pc,
  input                   i_rst_n,
  output [`INS_WIDTH-1:0] o_ins,
  output                  o_post_valid,
  input                   i_post_ready
);
  reg [`CPU_WIDTH-1:0] ins;

  // DPI-C 直接调用 C++ 内存读取 —— 组合逻辑，瞬时完成
  import "DPI-C" function void rtl_pmem_read(...);
  always @(*) begin
    rtl_pmem_read(i_pc, ins, i_rst_n);  // 一个 always 块就取完指令
  end

  assign o_post_valid = 1'b1;  // 永远有效（因为是瞬时的）
  assign o_ins = ins;
endmodule
```

**问题**：取指是组合逻辑瞬间完成的，不经过任何总线协议，与真实硬件行为不符。

---

### 14.2 改造后的 IFU（AXI-Lite Master，commit `32fd6d1`）

```verilog
module ifu (
  input                   i_clk,        // ← 新增: 需要时钟
  input                   i_rst_n,
  input  [`CPU_WIDTH-1:0] i_pc,
  output [`INS_WIDTH-1:0] o_instr,
  output                  o_post_valid,  // 取指完成信号
  input                   i_post_ready   // 下一级准备好了
);
  // ========== AXI-Lite 写通道 (IFU 不写, 全部置 0) ==========
  assign awaddr  = 0;
  assign awvalid = 0;
  assign wdata   = 0;
  assign wstrb   = 0;
  assign wvalid  = 0;
  assign bready  = 0;

  // ========== AXI-Lite 读通道 (核心逻辑) ==========
  assign araddr = i_pc;  // 读地址 = 当前 PC

  // ---------- 状态机: 管理 AR 通道握手 ----------
  // r_done_reg: 上一次读是否完成 (初始为1，表示可以开始新的读)
  // arvalid_reg: AR 通道的 valid 信号

  always @(*) begin
    // 默认保持
    r_done_next = r_done_reg;
    // 当下一级准备好接收 (ren=i_post_ready) 且上次已完成时, 发起新读请求
    if (ren) r_done_next = 1'b0;          // 标记: 读操作进行中
    // 当 R 通道握手成功时, 标记完成
    if (rvalid && rready) r_done_next = 1'b1;
  end

  always @(*) begin
    // arvalid 拉高后保持, 直到 arready 确认
    arvalid_next = arvalid_reg && !arready;
    // 上次完成 + 新请求到来 → 拉高 arvalid
    if (r_done_reg && ren) arvalid_next = 1'b1;
  end

  // 寄存器: 在时钟沿更新状态
  stdreg #(.WIDTH(2), .RESET_VAL({1'b1, 1'b0})) u_read_reg (
    .i_clk(i_clk), .i_rst_n(i_rst_n), .i_wen(1'b1),
    .i_din ({r_done_next, arvalid_next}),
    .o_dout({r_done_reg,  arvalid_reg})
  );

  assign rready = 1'b1;  // Master 随时准备接收读数据

  // ========== 内置 AXI-Lite SRAM Slave ==========
  axi_lite_sram u_axi_lite_sram (
    .awaddr(0), .awvalid(0), ...     // 写通道不用
    .araddr(araddr), .arvalid(arvalid_reg), .arready(arready),
    .rdata(rdata), .rvalid(rvalid), .rready(rready)
  );

  // ========== 输出 ==========
  assign o_instr      = rdata;              // 指令 = 从 SRAM 读回的数据
  assign o_post_valid = rvalid && rready;   // 取指完成 = R 通道握手成功
endmodule
```

---

### 14.3 IFU AXI-Lite 读操作时序

```
           cycle1     cycle2     cycle3
clock:   ──┐  ┌──┐  ┌──┐  ┌──┐  ┌──
           │  │  │  │  │  │  │  │

arvalid: ──────────┐           │
                   └───────────┘  (arready=1 后撤回)

arready: ──────────────────┐   │
                           └───┘  (Slave 接受地址)

araddr:  ──── PC值 ──────────────

rvalid:  ────────────────────────┐
                                 └── (数据就绪)

rdata:   ────────────────────────── 指令值 ──

o_post_valid: ──────────────────────────┐
                                        └── (取指完成!)
```

**关键**：从发出请求到得到数据，至少需要 **2 个时钟周期**（1拍发地址 + 1拍返数据）。如果 SRAM 有延迟，则需要更多。

---

### 14.4 改造前的 LSU（DPI-C 版本）

```verilog
module lsu (...);
  // DPI-C 直接读写 —— 组合逻辑瞬时完成
  always @(*) begin
    rtl_pmem_write(waddr, wdata, wmask);  // Store: 瞬时写入
    rtl_pmem_read(raddr, rdata, ren);     // Load: 瞬时读出
  end
  assign o_post_valid = i_pre_valid;  // 总是立即完成
endmodule
```

---

### 14.5 改造后的 LSU（AXI-Lite Master）

LSU 比 IFU 复杂——它既要读（Load）又要写（Store）。

**写操作状态机** (AW + W + B 三通道):
```verilog
  // 当收到 Store 请求时:
  always @(*) begin
    w_done_next = w_done_reg;
    if (wen) w_done_next = 1'b0;          // Store 开始 → 标记未完成
    if (bvalid && bready) w_done_next = 1'b1;  // B 通道确认 → 完成
  end

  always @(*) begin
    awvalid_next = awvalid_reg && !awready;
    wvalid_next  = wvalid_reg && !wready;
    if (w_done_reg && wen) begin           // 新 Store 请求
      awvalid_next = 1'b1;                 // 发起 AW
      wvalid_next  = 1'b1;                 // 同时发起 W
    end
  end
```

**读操作状态机** (AR + R 两通道)：与 IFU 相同逻辑。

**完成信号**:
```verilog
  // LSU 完成的三种情况:
  assign o_post_valid =
    (!w_done_reg && bvalid && bready) ||   // Store 刚完成
    (!r_done_reg && rvalid && rready) ||   // Load 刚完成
    (w_done_reg && r_done_reg && valid_r); // 非访存指令直通
```

**wstrb 字节使能生成**:
```verilog
  // 根据 lsu_opt 生成字节使能
  MuxKeyWithDefault u_wmask_mux (
    .key(lsu_opt),
    .lut({
      `LSU_SB, 4'b0001,   // Store Byte: 只写 1 字节
      `LSU_SH, 4'b0011,   // Store Half: 写 2 字节
      `LSU_SW, 4'b1111    // Store Word: 写 4 字节
    })
  );
```

---

### 14.6 AXI-Lite SRAM Slave 实现

Slave 端负责响应 Master 的请求，内部仍通过 DPI-C 访问主机内存：

```verilog
module axi_lite_sram (...);
  // ===== 读操作 =====
  // 条件: arvalid=1 且当前无未完成的读
  always @(*) begin
    if (arvalid && !arready && (!rvalid || rready)) begin
      mem_ren = 1'b1;             // 触发内存读取
      arready_next = 1'b1;       // 下一拍确认地址
      rvalid_next = 1'b1;        // 下一拍数据有效
    end
  end
  // DPI-C 读内存
  always @(*) rtl_pmem_read(araddr, rdata_next, mem_ren);

  // ===== 写操作 =====
  // 条件: awvalid=1 且 wvalid=1 (地址和数据都就绪)
  always @(*) begin
    if (awvalid && wvalid && (!awready && !wready) && (!bvalid || bready)) begin
      mem_wen = 1'b1;             // 触发内存写入
      awready_next = 1'b1;       // 下一拍确认地址
      wready_next = 1'b1;        // 下一拍确认数据
      bvalid_next = 1'b1;        // 下一拍发出响应
    end
  end
  // DPI-C 写内存
  always @(*) rtl_pmem_write(awaddr, wdata, wstrb, mem_wen);
endmodule
```

**Slave 的行为**：收到请求 → 下一拍返回结果（固定 1 拍延迟）。后续 v1.1/v1.2 中会加入随机延迟。

---

### 14.7 完整改造步骤总结

```
Step 1: 给 IFU 和 LSU 添加时钟输入 (i_clk)
        因为 AXI-Lite 是时序协议，需要寄存器存储状态

Step 2: 实现 IFU 的 AR/R 通道状态机
        - arvalid: 有请求时拉高，arready 确认后撤回
        - rready: 常高 (Master 总是准备好接收)
        - o_post_valid = rvalid && rready

Step 3: IFU 的 AW/W/B 通道全部置 0
        (IFU 只读不写)

Step 4: 实现 LSU 的 AR/R 通道 (Load)
        与 IFU 类似

Step 5: 实现 LSU 的 AW/W/B 通道 (Store)
        - awvalid + wvalid 同时拉高
        - 等待 bvalid 确认写完成
        - wstrb 根据 SB/SH/SW 设置字节使能

Step 6: 实现 AXI-Lite SRAM Slave
        - 接收 Master 请求
        - 通过 DPI-C 读写内存
        - 下一拍返回结果

Step 7: 连接 IFU/LSU 和 SRAM Slave

Step 8: 验证
        - 跑 cpu-tests + DiffTest
        - 查看波形确认握手时序正确
        - 加入随机延迟验证鲁棒性
```

---

### 14.8 验证：加入 LFSR 随机延迟

v1.2 中 SRAM Slave 的 ready 不再每次都在下一拍拉高，而是由 LFSR 决定延迟几拍：

```verilog
// 伪代码
if (arvalid && delay_counter == 0) begin
  delay_counter <= lfsr_value;  // 随机延迟 1~N 拍
end
if (delay_counter > 0) begin
  delay_counter <= delay_counter - 1;
end
if (delay_counter == 1) begin
  arready_next = 1'b1;  // 延迟结束，返回数据
  rvalid_next = 1'b1;
end
```

**目的**：如果你的状态机有 bug（比如没有正确等待 ready），在固定 1 拍延迟下可能恰好蒙对。加入随机延迟后，bug 就会暴露。

**终极验证**：在 LFSR 随机延迟下成功启动 RT-Thread → 说明 AXI-Lite 实现正确。

---

### 14.9 关键注意事项

1. **valid 一旦拉高不能撤回**（除非 ready 确认）：
   ```verilog
   arvalid_next = arvalid_reg && !arready;  // 只有 arready 才能清除
   ```

2. **AW 和 W 可以同时发起也可以分开**：本项目选择同时发起（简化逻辑）

3. **非访存指令的 LSU 处理**：当 `lsu_opt == LSU_NOP` 时，LSU 不发起任何 AXI 请求，直接通过：
   ```verilog
   assign o_post_valid = ... || (w_done_reg && r_done_reg && valid_r);
   ```

4. **地址需要寄存一拍**：因为 EXU 的结果是组合逻辑输出，可能不稳定。LSU 用 `stdreg` 将地址和数据锁存后再发到 AXI 总线：
   ```verilog
   stdreg u_w_keep_reg (..., .i_din({i_addr, i_regst}), .o_dout({awaddr, wdata}));
   ```

5. **IFU 的 PC 不需要寄存**：因为 PC 本身就是寄存器输出（BRU 在上一拍已经计算好了）

## 十五、详解：实现 AXI4-Lite 接口的 UART 功能

> 本节基于项目实际代码 (commit `8dc82bc` 的 `axi_lite_uart.v`)，详细展示如何将仿真环境中的串口功能移动到 RTL。

---

### 15.1 目标

之前：CPU 写 UART 地址时通过 DPI-C 回调到 C++ 的 `putc()` 输出字符。
现在：在 RTL 中实现一个 AXI-Lite Slave 模块，当收到写请求时用 `$write()` 输出字符。

```
之前 (DPI-C):                        现在 (RTL UART Slave):
  CPU (sw) → DPI-C pmem_write()        CPU (sw) → AXI-Lite Bus → Xbar
           → C++ putc()                            → axi_lite_uart.v
                                                    → $write("%c", data[7:0])
```

---

### 15.2 模块接口设计

```verilog
module axi_lite_uart (
  input  i_clk, i_rst_n,

  // AXI-Lite Slave 接口 (标准 5 通道)
  // AW Channel
  input  [31:0] awaddr,   input  awvalid,  output awready,
  // W Channel
  input  [31:0] wdata,    input  [3:0] wstrb,  input wvalid, output wready,
  // B Channel
  output [1:0]  bresp,    output bvalid,   input  bready,
  // AR Channel
  input  [31:0] araddr,   input  arvalid,  output arready,
  // R Channel
  output [31:0] rdata,    output [1:0] rresp, output rvalid, input rready
);
```

**说明**：接口与 `axi_lite_sram` 完全相同——这是总线的核心优势：**接口标准化**。任何 AXI-Lite Slave 都可以无缝替换。

---

### 15.3 内部设备寄存器

UART 模块内部只需要一个 32-bit 寄存器：

```verilog
wire [`CPU_WIDTH-1:0] uart_data_reg;  // 地址偏移: 0x0
reg  [`CPU_WIDTH-1:0] uart_data_next;
```

- 地址映射: 串口设备地址为 `0xa0000000`
- Xbar 将 `0xa0000000~0xa0000fff` 范围的请求路由到本模块
- 模块内部用 `awaddr[11:2]` 选择寄存器 (只有一个，偏移 0)

---

### 15.4 写操作实现 (AW + W + B 通道)

**当 CPU 执行 `sb char, 0(uart_addr)` 时**:

```verilog
// 写状态机 — 与 axi_lite_sram 完全相同的握手逻辑
always @(*) begin
  reg_wen = 1'b0;
  awready_next = 1'b0;
  wready_next = 1'b0;
  bvalid_next = bvalid_reg && !bready;  // bvalid 保持直到 bready 确认

  // 写握手条件: AW 和 W 同时有效，且当前无未完成事务
  if (awvalid && wvalid && (!awready && !wready) && (!bvalid || bready)) begin
    difftest_skip();         // ← 关键! UART 是设备访问，DiffTest 需跳过
    reg_wen = 1'b1;          // 触发设备寄存器写入
    awready_next = 1'b1;     // 下一拍确认地址
    wready_next = 1'b1;      // 下一拍确认数据
    bvalid_next = 1'b1;      // 下一拍返回写响应
  end
end
```

**设备寄存器写入** (带字节使能):
```verilog
always @(*) begin
  uart_data_next = uart_data_reg;
  if (reg_wen) begin
    case (awaddr[11:2])       // 地址解码 (仅 1 个寄存器)
      10'h0:                  // 偏移 0 = UART 数据寄存器
        for (byte_index = 0; byte_index <= 3; byte_index = byte_index + 1)
          if (wstrb[byte_index] == 1'b1)
            uart_data_next[(byte_index*8)+:8] = wdata[(byte_index*8)+:8];
      default: ;              // 忽略其他地址
    endcase
  end
end
```

**关键**: `wstrb` 字节使能处理！`putch('A')` 实际上是 `sb` 指令，只写 1 字节。`wstrb = 4'b0001` 时只更新最低字节。

---

### 15.5 字符输出 (核心功能)

```verilog
// 延迟一拍的写使能信号 (确保数据已写入 uart_data_reg)
wire sim_disp_en;
stdreg #(.WIDTH(1), .RESET_VAL(1'b0)) u_sim_out_reg (
  .i_clk(i_clk), .i_rst_n(i_rst_n), .i_wen(1'b1),
  .i_din(reg_wen), .o_dout(sim_disp_en)
);

// 当写使能有效时，输出低 8 位作为字符
always @(sim_disp_en) begin
  if (sim_disp_en) begin
    $write("%c", uart_data_reg[7:0]);  // ← 核心: 输出字符到终端
  end
end
```

**为什么要延迟一拍？**
- `reg_wen` 在当前周期有效时，`uart_data_reg` 还是旧值（新值要到下一个时钟沿才写入）
- 延迟一拍后，`sim_disp_en` 有效时 `uart_data_reg` 已经是新值
- 此时用 `$write` 输出才是正确的字符

---

### 15.6 读操作实现 (AR + R 通道)

UART 读操作不常用（真实 UART 可能读状态），但接口必须实现：

```verilog
always @(*) begin
  reg_ren = 1'b0;
  arready_next = 1'b0;
  rvalid_next = rvalid_reg && !rready;
  if (arvalid && !arready && (!rvalid || rready)) begin
    difftest_skip();         // UART 读也要跳过 DiffTest
    reg_ren = 1'b1;
    arready_next = 1'b1;
    rvalid_next = 1'b1;
  end
end

// 根据地址返回对应寄存器值
always @(*) begin
  case (araddr[11:2])
    10'h0:   rdata_next = uart_data_reg;   // 读 UART 数据寄存器
    default: rdata_next = 0;
  endcase
end
```

---

### 15.7 接入 Xbar

实现 UART 后需要将其接入总线系统。需要一个 **Xbar (Crossbar/地址解码器)**:

```verilog
// top.v 中 (commit 8dc82bc)
// Xbar 将不同地址范围路由到不同 Slave:
//   0x80000000~0x87ffffff → SRAM (主存)
//   0xa0000000~0xa0000fff → UART
//   0xa0002000~0xa0002fff → CLINT (定时器)
```

Xbar 的核心逻辑就是**地址比较**：
```verilog
// 伪代码
if (addr >= 0xa0000000 && addr < 0xa0001000)
  route_to = UART;
else if (addr >= 0xa0002000 && addr < 0xa0003000)
  route_to = CLINT;
else
  route_to = SRAM;
```

项目中实现为 `axi_lite_xbar.v` (439 行)，完整处理了多 Slave 的地址解码和信号多路复用。

---

### 15.8 DiffTest Skip 的关键性

```verilog
// 在写握手条件满足时调用
difftest_skip();
```

**为什么必须 skip？** 因为：
- NPC (DUT) 写 UART 后通过 `$write` 输出字符
- NEMU (REF) 写 UART 后通过 C 的 `putc` 输出字符
- 两者的 UART"设备状态"不一致（NPC 是 RTL 寄存器，NEMU 是 C 变量）
- 如果不 skip，DiffTest 对比时会发现寄存器状态不同（因为 UART 写入可能影响后续指令的行为）

---

### 15.9 完整实现步骤

```
Step 1: 编写 axi_lite_uart.v
        - 复制 axi_lite_sram.v 的握手框架
        - 删除内存相关逻辑
        - 添加一个 uart_data_reg
        - 写入时用 $write 输出低 8 位

Step 2: 编写 axi_lite_xbar.v (或简化版地址解码器)
        - 输入: 来自 Arbiter 的 AXI-Lite 请求
        - 根据地址范围选择输出端口 (SRAM / UART / CLINT)
        - 将请求转发给对应 Slave
        - 将 Slave 响应回传给 Master

Step 3: 修改 top.v 连接
        - Arbiter 输出 → Xbar 输入
        - Xbar 端口0 → SRAM
        - Xbar 端口1 → UART
        - (可选) Xbar 端口2 → CLINT

Step 4: 修改地址映射
        - AM 中 SERIAL_PORT = 0xa0000000
        - Xbar 中 UART 地址范围 = 0xa0000000~0xa0000fff

Step 5: 移除 DPI-C 中的 UART 处理
        - dpic.cpp 中不再需要 if(addr==0xa0000000) putc(...)
        - 改由 RTL 的 $write 输出

Step 6: 验证
        - 运行 hello 程序，确认输出 "Hello, AbstractMachine!"
        - 运行 cpu-tests，确认功能正确
        - 开启 DiffTest，确认 skip 正确工作
```

### 15.10 从 UART 到 CLINT 的推广

实现 UART 后，用同样的模式实现 CLINT (定时器) 就很简单了:

| 方面 | UART | CLINT |
|------|------|-------|
| 地址 | 0xa0000000 | 0xa0002000 |
| 寄存器数 | 1 个 (数据) | 2 个 (mtime_lo, mtime_hi) |
| 核心操作 | 写 → `$write` 输出 | 读 → 返回 `$time` |
| DiffTest | 写时 skip | 读时 skip |

这就是 commit `8dc82bc` 中同时添加 UART 和 CLINT 的原因——它们的 AXI-Lite Slave 框架完全相同，只是内部寄存器行为不同。

## 十六、详解：实现 AXI4-Lite 接口的 CLINT

> 本节基于项目实际代码 (commit `8dc82bc` 的 `axi_lite_clint.v`)。

---

### 16.1 CLINT 功能定义

CLINT (Core Local INTerrupt controller) 在当前阶段只实现时钟功能：

- **mtime 寄存器**: 64-bit，每时钟周期自增 1
- **地址**: `0xa0002000` (低 32 位), `0xa0002004` (高 32 位)
- **只读**: CPU 只能读取，不能写入（写通道全部置 0）

---

### 16.2 与 UART 的关键区别

| 方面 | UART | CLINT |
|------|------|-------|
| 核心操作 | **写**（输出字符） | **读**（获取时间） |
| 写通道 | 完整实现 | 全部置 0 (只读设备) |
| 寄存器位宽 | 32-bit | **64-bit** (分两次 32-bit 读) |
| 自更新 | 无 | 每周期 +1 |
| DiffTest Skip | 写时 skip | **读时 skip** |

---

### 16.3 完整实现代码解析

#### 16.3.1 写通道 — 全部禁用

```verilog
// CLINT 是只读设备，写通道信号全部置 0
assign awready = 1'b0;   // 永不接受写地址
assign wready  = 1'b0;   // 永不接受写数据
assign bvalid  = 1'b0;   // 永不返回写响应
assign bresp   = 2'b00;
```

**效果**：如果 CPU 误对 CLINT 地址执行 Store 指令，AW/W 通道永远不会握手成功，CPU 会卡住。在仿真中可以通过 assert 检测这种情况。

#### 16.3.2 mtime 计数器 — 每周期 +1

```verilog
wire [`CPU_WIDTH*2-1:0] mtime_reg;   // 64-bit 计数器
reg  [`CPU_WIDTH*2-1:0] mtime_next;

// 每周期自增
always @(*) begin
  mtime_next = mtime_reg + 1'b1;
end

// 64-bit 寄存器 (复位值 = 0)
stdreg #(
  .WIDTH    (`CPU_WIDTH * 2),         // 64 位!
  .RESET_VAL({(`CPU_WIDTH * 2){1'b0}})
) u_mtime_reg (
  .i_clk  (i_clk),
  .i_rst_n(i_rst_n),
  .i_wen  (1'b1),        // 永远使能写入 (每拍都更新)
  .i_din  (mtime_next),
  .o_dout (mtime_reg)
);
```

**关键设计**：
- 寄存器宽度为 64 位 (防止溢出)
- `i_wen = 1'b1` — 每个时钟周期都写入新值 (mtime 永远在增长)
- 在 500MHz 下，64-bit 计数器需要 ~584 年才会溢出

#### 16.3.3 读操作 — 分低/高 32 位

```verilog
// 地址解码: 根据 araddr[11:2] 选择返回哪半部分
always @(*) begin
  case (araddr[11:2])                         // 字对齐地址 (除以 4)
    10'h0:   rdata_next = mtime_reg[31:0];    // 偏移 0x0 → 低 32 位
    10'h1:   rdata_next = mtime_reg[63:32];   // 偏移 0x4 → 高 32 位
    default: rdata_next = `CPU_WIDTH'b0;
  endcase
end
```

**为什么用 `araddr[11:2]`？**
- `[1:0]` 是字节偏移（对于 32-bit 总线无意义）
- `[11:2]` 是字偏移（寄存器编号）
- 偏移 0x0 (`araddr[11:2] = 10'h0`) → mtime 低 32 位
- 偏移 0x4 (`araddr[11:2] = 10'h1`) → mtime 高 32 位

#### 16.3.4 读握手状态机

```verilog
always @(*) begin
  reg_ren = 1'b0;
  arready_next = 1'b0;
  rvalid_next = rvalid_reg && !rready;   // rvalid 保持直到 rready 确认

  if (arvalid && !arready && (!rvalid || rready)) begin
    difftest_skip();       // 读时间 = 设备访问，必须 skip
    reg_ren = 1'b1;        // 锁存当前 rdata_next 到 rdata_reg
    arready_next = 1'b1;   // 下一拍确认地址
    rvalid_next = 1'b1;    // 下一拍数据有效
  end
end
```

**时序**:
```
cycle N:    arvalid=1, araddr=0xa0002000
cycle N+1:  arready=1, rvalid=1, rdata=mtime[31:0]  ← 数据返回
cycle N+2:  (CPU 读到数据)
```

### 16.4 软件端的配合修改

#### 16.4.1 AM timer 实现 (`abstract-machine/am/src/riscv/npc/timer.c`)

```c
#define RTC_ADDR 0xa0002000

static uint64_t get_time() {
    uint32_t lo = inl(RTC_ADDR);       // 读低 32 位 (地址 0xa0002000)
    uint32_t hi = inl(RTC_ADDR + 4);   // 读高 32 位 (地址 0xa0002004)
    return ((uint64_t)hi << 32) + lo;
}

void __am_timer_uptime(AM_TIMER_UPTIME_T *uptime) {
    uptime->us = get_time() - boot_time;
}
```

#### 16.4.2 时间系数处理

mtime 每周期 +1，但 AM 返回的是**微秒**。需要一个系数转换：

```c
// 如果仿真中 1 个周期 ≈ 1 微秒 (取决于仿真速率)
uptime->us = get_time() - boot_time;

// 如果在真实 FPGA 上跑 100MHz (1 周期 = 10ns = 0.01 微秒)
uptime->us = (get_time() - boot_time) / 100;
```

项目中直接使用 `get_time() - boot_time`（假设系数为 1），因为仿真环境的"时间"就是周期数。

---

### 16.5 64-bit 读取的原子性问题

**问题**: CPU 分两次读 mtime 的低/高 32 位，但 mtime 在两次读之间可能进位！

```
例: mtime = 0x00000000_FFFFFFFF (马上要进位)
  CPU 读低 32 位: lo = 0xFFFFFFFF
  (此时 mtime 进位变成 0x00000001_00000000)
  CPU 读高 32 位: hi = 0x00000001
  组合结果: 0x00000001_FFFFFFFF ← 错! 应该是 0x00000000_FFFFFFFF 或 0x00000001_00000000
```

**解决方案** (软件层面):
```c
uint64_t get_time_safe() {
    uint32_t hi1, lo, hi2;
    do {
        hi1 = inl(RTC_ADDR + 4);   // 读高位
        lo  = inl(RTC_ADDR);       // 读低位
        hi2 = inl(RTC_ADDR + 4);   // 再读高位
    } while (hi1 != hi2);          // 如果高位变了，重试
    return ((uint64_t)hi1 << 32) + lo;
}
```

**实际影响**: 对于仿真环境（每秒执行几十万条指令），32-bit 溢出需要 ~4294 秒 (>1小时)，通常不会遇到。但真实 500MHz 硬件上 32-bit 约 8.6 秒就会溢出，必须处理。
